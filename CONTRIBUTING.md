# Contributing to Mycelix Protocol

Thank you for your interest in contributing to the Mycelix Protocol! This document provides guidelines for contributing to this landing page and the broader project.

This repository contains the **landing page** for mycelix.net, which showcases the PoGQ (Proof of Gradient Quality) innovation. The main protocol implementation is in the [Luminous-Dynamics/mycelix](https://github.com/Luminous-Dynamics/mycelix) repository.

## ü§ù How to Contribute

### Landing Page Improvements

We welcome contributions that improve:

1. **Accessibility** - WCAG 2.1 compliance, screen reader support
2. **Performance** - Load times, rendering optimization
3. **SEO** - Search engine visibility, metadata improvements
4. **Content** - Clarity, accuracy, technical correctness
5. **Design** - Visual polish, responsive behavior, UX

### Getting Started

1. **Fork the repository**
   ```bash
   git fork https://github.com/Luminous-Dynamics/mycelix.net
   ```

2. **Clone your fork**
   ```bash
   git clone https://github.com/YOUR-USERNAME/mycelix.net.git
   cd mycelix.net
   ```

3. **Create a feature branch**
   ```bash
   git checkout -b feature/your-improvement-name
   ```

4. **Make your changes**
   - Edit `index.html` for content/design changes
   - Update `README.md` if you add new features
   - Test locally (open `index.html` in a browser)

5. **Commit your changes**
   ```bash
   git add .
   git commit -m "feat: brief description of your improvement"
   ```

6. **Push to your fork**
   ```bash
   git push origin feature/your-improvement-name
   ```

7. **Create a Pull Request**
   - Go to the original repository
   - Click "New Pull Request"
   - Select your fork and branch
   - Describe your changes clearly

## üìù Commit Message Guidelines

We follow the [Conventional Commits](https://www.conventionalcommits.org/) specification:

- `feat:` - New features or enhancements
- `fix:` - Bug fixes
- `docs:` - Documentation changes
- `style:` - Code style/formatting (no functional changes)
- `refactor:` - Code refactoring
- `perf:` - Performance improvements
- `test:` - Adding or updating tests
- `chore:` - Maintenance tasks

Examples:
```
feat: add FAQ section to landing page
fix: improve mobile responsiveness on stats cards
docs: update deployment instructions in README
perf: optimize CSS and reduce file size
```

## üß™ Testing Checklist

Before submitting a PR, please verify:

- [ ] HTML validates ([W3C Validator](https://validator.w3.org/))
- [ ] CSS validates ([CSS Validator](https://jigsaw.w3.org/css-validator/))
- [ ] Accessibility tested ([WAVE](https://wave.webaim.org/))
- [ ] Mobile responsive (test on different screen sizes)
- [ ] Cross-browser compatible (Chrome, Firefox, Safari, Edge)
- [ ] All links work correctly
- [ ] No console errors

## üé® Design Guidelines

- **Colors**: Maintain the purple gradient theme (#667eea, #764ba2)
- **Typography**: Use system fonts for performance
- **Accessibility**: Maintain WCAG 2.1 AA compliance minimum
- **Performance**: Keep page load under 2 seconds
- **Mobile-first**: Design for mobile, enhance for desktop

## üîç Code Review Process

1. All PRs require review before merging
2. Reviewers will check for:
   - Code quality and standards
   - Accessibility compliance
   - Performance impact
   - Browser compatibility
3. Address feedback promptly
4. Once approved, maintainers will merge

## üêõ Reporting Issues

Found a bug or have a suggestion?

1. Check [existing issues](https://github.com/Luminous-Dynamics/mycelix.net/issues)
2. If not found, [create a new issue](https://github.com/Luminous-Dynamics/mycelix.net/issues/new)
3. Provide:
   - Clear description
   - Steps to reproduce (for bugs)
   - Expected vs actual behavior
   - Browser/device information
   - Screenshots if applicable
